# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/), and this project adheres to [Semantic Versioning](https://semver.org/).

## [Unreleased]

## [0.0.18] - 2026-04-21

### Changed

- **Slim DevQL `selectArtefacts` now treats file and directory paths as distinct selection modes**: file paths continue to expose `artefacts` from the selected file, while directory paths now behave as immediate-child `ls` listings through `entries` only. Directory selectors now reject artefact-only fields such as `artefacts`, `summary`, `checkpoints`, `clones`, `deps`, and `tests`, and file selectors now reject `entries`.
- **`bitloops init --install-default-daemon` now stages semantic setup through explicit conditional lanes**: init progress is now reported as separate `Sync`, `Ingest`, `Code Embeddings`, `Summaries`, and `Summary Embeddings` lanes instead of combining sync and embedding work into broader sections. Runtime execution now follows the selected dependency order: sync runs first when enabled, ingest is queued after sync but remains independently selectable, code embeddings and summaries only run when sync is selected, and summary embeddings only run when summaries are enabled. During the summaries prompt, `Skip for now (recommended)` is now the default preselected option and `Bitloops Cloud (limited availability)` is listed second.
- **Plain `bitloops init` now follows the same onboarding shape as the default-daemon flow, minus daemon-owned setup**: when embeddings and semantic summaries are still unconfigured, interactive plain init no longer prompts for Bitloops Cloud vs local embeddings, no longer asks to configure semantic summaries, and no longer performs those daemon-owned setup actions. Existing configured embeddings or summary lanes still participate in init runtime work, unresolved telemetry consent still appears in the final setup step, and the daemon auto-start option remains exclusive to `--install-default-daemon`.

### Added

- **Slim `selectArtefacts` now supports typo-tolerant fuzzy artefact lookup by symbol name**: slim GraphQL requests can now select current artefacts with `by: { fuzzyName: "payLater()" }`, and the DevQL DSL now compiles `selectArtefacts(fuzzy_name:"payLater()")` to the same selector. The fuzzy matcher normalizes human-entered names from `symbol_fqn` leaf segments, preserves exact `symbolFqn` behaviour as an explicit selector, ranks matches best-first with exact/prefix/token/edit-distance signals, filters out weak matches below the fixed `0.6` threshold, and caps results at 10. The slim SDL snapshot, DevQL docs, and Bitloops-managed agent guidance/skill content now document the new selector mode.

### Fixed

- **DevQL hook bootstrap now defers to installed repo-local skills and rules**: shared agent hook augmentation no longer inlines `using-devql` skill content or emits hook-time `bitloops devql query` suggestions. Session-start bootstrap now appears only when the agent-specific repo-local DevQL skill or rule is actually installed and only points the agent at that surface, while turn-start hook augmentation stays silent. OpenCode now follows the same presence-only contract through its plugin bootstrap path, and `bitloops init --disable-bitloops-skill` now suppresses both repo-local DevQL surfaces and the corresponding session-start bootstrap messaging.

## [0.0.17] - 2026-04-19

### Fixed

- **Semantic-clones repo-backfill follow-up batches no longer rehydrate the entire current repository view**: explicit `artefact_ids` carried by summary-refresh and embedding mailbox follow-up batches now use a targeted current-projection loader instead of rebuilding semantic inputs for every current file in the repo before filtering. This removes repeated full-repo hydration during large backfills, cuts unnecessary `git cat-file` churn for `head`-backed current paths, and keeps unrelated broken or missing current files from failing explicit follow-up batches that do not need them.

## [0.0.16] - 2026-04-17

### Added

- **Dashboard interaction subscriptions for live refresh**: the dashboard GraphQL surface now exposes `interactionUpdates` over `/devql/dashboard/ws`, including repo-scoped session and turn counts plus the latest session/turn ids and update timestamps. The dashboard subscription stream uses a lightweight interaction-spool change snapshot so clients can automatically refresh when new sessions or turns arrive without falling back to full-page polling.

### Changed

- **DevQL sync now stops at capability-event enqueue instead of waiting for semantic-clones projection**: sync now completes after current-state writes, touched-path local-edge reconciliation, and capability-event enqueue, and no longer runs semantic-clones projection, embedding refresh, or clone rebuild inline from the sync orchestrator. Semantic-clones current rows may now be briefly empty or stale for affected paths until the deferred pipeline drains, while sync observability now splits `materialisation_total`, `current_edge_reconcile_total`, and `capability_event_enqueue_total` so downstream semantic backlog is no longer reported as sync time.

### Fixed

- **Queued SQLite current-edge reconciliation now survives very large sync path sets**: the post-sync local dependency reconciliation path now loads touched paths into a temporary SQLite table and joins resolved current edges against canonical `to_symbol_ref` paths instead of building one `LIKE ... OR ...` predicate per touched file. This preserves the existing queue-driven sync flow while preventing `Expression tree is too large (maximum depth 1000)` failures on large repositories such as Ruff.
- **`bitloops init` now keeps repo-managed agent skill files out of git status**: after resolving the selected agents, init now adds the installed repo-local DevQL skill paths for Claude Code, Codex, Gemini CLI, Copilot CLI, and OpenCode to `.git/info/exclude` alongside `.bitloops.local.toml`. This prevents the managed skill files from showing up as untracked changes in repos that bootstrap agent guidance through `bitloops init`.
- **Semantic-clones deferred repo backfill now executes through the existing workplane mailboxes**: `semantic_clones` current-state reconciliation now owns the sync-to-semantic scheduling path, clears stale current semantic and embedding rows for affected paths before enqueueing deferred work, and uses the existing summary-refresh, embedding, and clone-rebuild mailboxes for both per-artefact deltas and full-reconcile repo backfills. Repo-backfill summary-refresh and embedding jobs now load current-repo inputs instead of returning early, so deferred full reconciles and backlog-compacted work actually repopulate current semantic features, current embeddings, and clone rebuild follow-ups.
- **Current local-edge reconciliation no longer falls back to repo-wide scans for touched target updates**: the sync-side reconciliation pass now scopes current target and source-fact loading to touched targets and their dependent source paths, adds the supporting current-state indexes for `artefact_edges_current` and `artefacts_current`, and still preserves late-resolution behaviour when touched targets appear, disappear, or force previously unresolved edges to be recomputed. This restores the intended fast sync path on larger repositories without regressing grouped Rust imports or other cross-file local-edge resolution cases.
- **Sync completion now persists scope-exclusion state before reporting success**: DevQL sync now records the resolved scope-exclusions fingerprint as part of the same completion write that marks `repo_sync_state` as `completed`, so the daemon no longer sees a transient â€ścompleted sync but missing exclusions fingerprintâ€ť state. This prevents a spurious hidden `repo_policy_change` reconcile sync from being enqueued immediately after a normal sync, which in turn stops queued ingest work from sitting behind that unnecessary follow-up sync during init and other sync-then-ingest flows.
- **Interaction GraphQL date filters now treat equivalent UTC offsets as the same instant**: interaction browse and search filters no longer fall back to raw string ordering after successfully parsing RFC 3339 timestamps. This fixes inclusive `since` / `until` boundaries when GraphQL normalises `Z` inputs to `+00:00`, restoring slim and global `interactionSessions` queries that would otherwise drop matching sessions at the time-window edge.
- **Codex interaction and checkpoint persistence now survive missing `Stop.transcriptPath` payloads**: the lifecycle-routed Codex hook path now resolves real Codex rollout transcripts from saved pre-prompt/session state and the date-sharded `~/.codex/sessions` store before failing turn-end capture. This restores persistence of `interaction_sessions`, `interaction_turns`, `interaction_events`, and checkpoint save-step data for Codex sessions, keeps later turns scoped to their new transcript fragment instead of replaying from offset `0`, and teaches transcript metadata extraction to read real Codex `response_item` payloads for prompt and summary capture.
- **Lifecycle token usage now persists for Claude Code, Codex, and OpenCode sessions**: Claude Code and OpenCode lifecycle routing now expose their existing transcript-backed token calculators, so turn-end capture can persist token usage into checkpoint save-step metadata, `interaction_turns`, and the downstream Event DB sync instead of leaving relational and event-side token fields empty. Codex token accounting now computes per-turn deltas from cumulative `event_msg` `token_count` records, separates cached prompt tokens from regular input tokens, and counts multiple model calls within the same turn correctly. Gemini CLI and Copilot CLI remain covered by lifecycle token capture, while Cursor still cannot emit token usage until upstream adds token counters to its local session artefacts.
- **Configured semantic summaries no longer silently fall back to template-only current projection rows**: current-path sync projection and unchanged-repo current-state rehydration now resolve `summary_generation` in strict mode when a local or cloud summary provider is configured. This prevents Bitloops from persisting deterministic/template-only summaries in `symbol_semantics_current` when summary generation is expected, and turns misconfigured current-projection summary providers into explicit sync failures instead of silent degradation. Deterministic fallback summaries still apply when no summary provider is configured.
- **Canonical semantic summaries now preserve non-empty LLM output regardless of length**: semantic summary promotion no longer rejects model-backed summaries once they cross the old 200-character cap. When an `llm_summary` exists, Bitloops now keeps it in the canonical `summary` instead of degrading back to template-only text, and deterministic fallback only applies when no LLM summary is available.
- **Init enrichment progress now tracks repo coverage instead of compacted workplane job counts**: the init progress screen now measures code-embedding and semantic-summary progress from repo-scoped persisted coverage, so repo-backfill jobs no longer render misleading tiny denominators such as `0/2 artefacts`. The summary lane also re-checks repo queue state after setup completes, which prevents it from staying marked complete when later sync or ingest phases enqueue real summary work.

## [0.0.15] - 2026-04-16

### Added

- **Repeatable init agent selection**: `bitloops init` now accepts `--agent` multiple times so one bootstrap command can configure multiple supported agent surfaces. Explicit CLI agent names are normalised through the registry, deduplicated after normalisation, and then persisted through the existing repo-policy and hook-reconciliation flow.
- **Multi-agent DevQL hook augmentation bootstrap**: Bitloops now emits agent-aware DevQL bootstrap context across the supported hook surfaces instead of treating every agent like Claude. Claude Code, Codex, Gemini CLI, and Copilot CLI receive model-visible `SessionStart` guidance and prompt-start guidance through their native hook-output fields; OpenCode receives the same guidance through the repo-local plugin message transform path; Cursor continues to receive session-start bootstrap guidance through its current hook path. The injected guidance remains capability-only: it teaches agents how to use `bitloops devql query ...` and `bitloops devql schema ...`, but it does not pre-run DevQL or inject live query results.
- **Native repo-local DevQL guidance assets for supported agents**: Bitloops now installs a native repo-local DevQL instruction surface for each supported agent instead of relying on bootstrap text alone. This includes `.claude/skills/bitloops/using-devql/SKILL.md` for Claude Code, `.agents/skills/bitloops/using-devql/SKILL.md` for Codex, `.github/skills/bitloops/using-devql/SKILL.md` for Copilot CLI, `.gemini/skills/bitloops/using-devql/SKILL.md` plus a managed `GEMINI.md` import block for Gemini CLI, `.opencode/skills/bitloops/using-devql/SKILL.md` for OpenCode, and `.cursor/rules/bitloops-using-devql.mdc` for Cursor. All of these surfaces are derived from the same canonical `using-devql` guidance so hook bootstrap text and installed repo artifacts stay aligned.
- **Project-aware DevQL file classification**: DevQL now classifies repository paths as `code`, `text`, `track_only`, or `excluded` instead of treating code relevance as a simple file-extension decision. Repositories can contain multiple project contexts, each file records one primary context plus any secondary matches, and the resolved classification now drives sync, ingest, watch, refresh, and current-state persistence.
- **Secondary file-role and text-index metadata for DevQL relevance**: classified files now also record a `file_role` such as `source_code`, `project_manifest`, `context_seed`, `configuration`, or `documentation`, plus a `text_index_mode` of `embed`, `store_only`, or `none`. This lets DevQL distinguish code relevance from text-indexing policy while keeping manifests and context/config files visible to project resolution and invalidation.
- **Manual repo-policy controls for relevance and exclusions**: repo policy now supports manual `[[contexts]]` declarations for positive code inclusion, `scope.include_as_text` for explicit text promotion, and repeatable `bitloops init --exclude <glob>` / `--exclude-from <path>` flags that are persisted to `.bitloops.local.toml` before any init-triggered sync or ingest work.
- **Automatic exclusion reconciliation for DevQL storage and queues**: the daemon now fingerprints the resolved repo-policy exclusion snapshot, persists the last applied exclusions fingerprint in `repo_sync_state`, and automatically enqueues one repo-scoped reconciliation sync when exclusions change. That reconciliation purges newly excluded current and historical path-scoped DevQL rows, clears stale current-state consumer backlog, prunes pending single-artefact enrichment jobs whose inputs disappeared, and only records the new fingerprint after the reconcile sync completes successfully.
- **Managed standalone `bitloops-inference` installation and IPC contract**: added `bitloops inference install`, managed download and repair flow for release-tagged `bitloops-inference` binaries, persisted install metadata and eligible daemon-config command rewrites, plus a shared `bitloops-inference-protocol` crate that defines the JSON `describe` / `infer` / `shutdown` contract between Bitloops and the standalone runtime.
- **CLI WorkOS sign-in with secure persisted sessions**: added `bitloops login`, `bitloops login status`, and `bitloops logout`. The CLI now uses WorkOS AuthKit device login, stores token material in the platform credential store, persists session metadata in the daemon runtime store, and refreshes access tokens automatically when the stored session is still valid.
- **Automatic local semantic-summary bootstrap**: interactive `bitloops enable` and `bitloops init --install-default-daemon` can now install or repair `bitloops-inference`, detect local Ollama availability, and bind semantic summaries to a default local text-generation profile when a suitable Ollama model is available. Added `BITLOOPS_INFERENCE_VERSION_OVERRIDE` for pinning managed installs to a specific runtime release.
- **Managed Bitloops platform embeddings runtime and cloud summary bootstrap**: Bitloops can now install and wire a hosted `platform` embeddings runtime through `bitloops embeddings install --runtime platform`, `bitloops init --embeddings-runtime platform`, and `bitloops enable --install-embeddings --embeddings-runtime platform`. The new managed `bitloops-platform-embeddings` binary keeps the same daemon-facing IPC contract as the local runtime, uses `BITLOOPS_PLATFORM_GATEWAY_TOKEN` by default for hosted gateway auth, and pairs with a default `summary_llm` semantic-summary profile when you choose the Bitloops cloud setup path.
- **Explicit request defaults for text-generation profiles**: text-generation profiles now support `temperature` and `max_output_tokens`, and the unified config model carries those values through profile resolution for runtime-backed summary generation.

### Changed

- **Breaking â€” DevQL indexing now follows project setup before extracting code**: Rust, Node/JavaScript, TypeScript/TSX, Python, Go, and Java files are now only treated as code when they belong to a detected or manually declared project context. Human-authored docs, manifests, and config files are indexed as `text`, while contextless code-like files default to `track_only` instead of being indexed as code. This also introduces persisted `file_role` / `text_index_mode` metadata in current-state DevQL tables, and schema mismatches now rebuild those sync tables instead of preserving legacy rows.
- **DevQL ingest orchestration was split into focused submodules**: `commands_ingest.rs` is now a thin entrypoint that delegates orchestration, semantic-refresh helpers, shared utilities, and progress emission into dedicated sibling modules. This keeps the ingest surface easier to navigate without changing the public CLI or GraphQL behavior.
- **DevQL bootstrap messaging is now agent-specific and repo-surface aware**: the hook augmentation builder no longer hardcodes Claude-specific skill references for every agent. Session-start bootstrap now points each agent at its own installed repo-local skill or rule surface, while turn-start guidance remains prompt-aware and command-oriented so the agent runs DevQL itself.
- **Repo-policy scope and exclusion inputs now affect classification and invalidation end to end**: `scope.project_root` and `scope.include` now constrain automatic relevance detection, `exclude_from` paths are normalised to the repo-policy root and rejected when they point outside it, and referenced exclusion files are re-read on each evaluation so their paths and contents participate in the repo-policy fingerprint.
- **Queued repo work now respects exclusion changes before execution**: queued path-scoped sync tasks now drop newly excluded paths and cancel themselves when nothing remains, pending producer-spool refresh jobs are pruned against the current exclusion matcher on recovery and claim, repo-policy reconciliation syncs take precedence over lower-priority repo work, and workplane enrichment jobs that reload zero inputs after reconciliation now exit cleanly instead of failing.
- **Breaking â€” text-generation profiles now run through configured inference runtimes**: `task = "text_generation"` profiles must now declare `runtime`, runtime commands may not be blank, and Bitloops no longer instantiates hosted LLM clients directly for semantic summaries. Instead it launches the configured runtime, with docs and examples updated to use `bitloops_inference`, `openai_chat_completions`, and the chat-completions endpoint for OpenAI-compatible providers.
- **Semantic summary execution now uses shared standalone runtime sessions**: Bitloops now reuses process-wide `bitloops-inference` sessions keyed by runtime configuration and profile, validates the runtime handshake before use, retries once after runtime failures, and evicts idle sessions after 60 seconds so summary generation stays warm without leaving stale processes running.
- **Semantic-summary onboarding is now cloud-aware and interactive**: interactive `bitloops init` / `bitloops enable` summary setup now offers Bitloops cloud as the recommended option, local Ollama as the on-device alternative, or skip for now. TTY sessions use a single-select picker, cloud setup opens the browser-based sign-in flow when needed, hosted chat-completions URLs can be derived from `BITLOOPS_PLATFORM_GATEWAY_URL`, and the default managed cloud summary model is now `ministral-3-3b-instruct`.
- **Managed local summary bootstrap now writes complete Ollama chat defaults**: auto-configured `summary_local` profiles now use the Ollama chat endpoint at `http://127.0.0.1:11434/api/chat` and persist default `temperature = "0.1"` plus `max_output_tokens = 200`, while config resolution and runtime startup now warn or fail fast when text-generation profiles omit required request parameters.
- **Documentation now recommends the single-command fast start path**: the README and getting-started guides now present `bitloops init --install-default-daemon` as the fastest way to get started, while keeping `bitloops start --create-default-config` documented for explicit or custom daemon bootstraps.

### Fixed

- **Managed runtime GitHub Releases downloads no longer buffer whole archives in memory during transfer**: `bitloops embeddings install`, managed platform embeddings installs, and managed `bitloops-inference` installs now stream release assets into a temporary file, compute the SHA-256 digest while bytes arrive, and extract from disk afterwards. This reduces peak memory use and removes the extra in-memory copy that was making larger runtime downloads slower than necessary.
- **Degraded code files now persist as file-only DevQL artefacts**: DevQL sync and historical ingest no longer drop code paths whose worktree bytes or git blob bytes fail UTF-8 decoding, and sync now also degrades code files that are valid UTF-8 but contain embedded NUL bytes or otherwise fail code extraction before semantic projection. Those paths now persist current/historical file state plus exactly one `file` artefact, still count toward `parse_errors`, and skip nested symbol extraction, dependency edges, summaries, features, and embeddings so the overall run degrades cleanly instead of failing or silently losing the file.
- **Historical ingest no longer silently skips unreadable git blobs**: when `git_blob_decoded_content(...)` cannot load blob bytes for a changed historical path, ingest now records a contextual failure for that commit instead of silently `continue`-ing past the path. This keeps degraded runs visible in the ingest ledger with path, commit, and blob context instead of dropping the file state update without a trace.
- **Codex `UserPromptSubmit` hook timeouts on large or dirty repositories**: the turn-start path no longer eagerly computes manual-commit prompt attribution during `initialize_session`, which was forcing the Codex prompt-submit hook to pay for working-tree scans and Git tree snapshot diffing before returning hook output. Prompt attribution now falls back at checkpoint-save time when needed, and the managed Codex hook config now keeps the general `10s` timeout for the other hooks while raising `UserPromptSubmit` to `30s`.
- **DevQL GraphQL end-to-end fixture stabilisation**: the Rust fixture used by the GraphQL smoke and verification tests is now generated as a valid crate with a manifest and module layout, preventing background current-state sync from rewriting the seeded artefact set into an incomplete view and keeping raw GraphQL, default GraphQL, and DSL query output aligned.
- **Semantic-clone real-path fixture completeness and expectations**: real-path BDD fixtures now persist summary embeddings alongside code embeddings, and the semantic-clone feature expectations were refreshed to match the classifier's explainable output, including the lower-threshold clone case and unrelated-pair suppression coverage.
- **Semantic-clones health coverage for summary generation**: capability health now validates `summary_generation` slot binding, expected task type, runtime presence, runtime command resolution, and live runtime handshakes alongside embeddings, so misconfigured semantic summary profiles fail fast instead of appearing healthy.
- **Standalone `bitloops-inference` startup without Bitloops cloud auth**: platform gateway bearer tokens are now injected into text-generation runtime launches only when a valid CLI session exists, so local Ollama and other non-platform summary profiles continue to start cleanly without requiring a Bitloops login.
- **Repo-scoped exclusion reconciliation now ignores daemon config directories that are not repositories**: the coordinator only seeds reconciliation from the current path when that path is itself a Git repository, and registry-driven reconciliation now keeps only repositories bound to the active daemon config. This prevents exclusion updates from spilling into unrelated daemon config roots or repos registered under a different binding.
- **Test-harness current-state parity for Rust linkage discovery**: automatic sync now schedules and materializes `test_harness.current_state` through the generic cursor queue, closes the Ruff-style static-linkage gap, and documents legacy `ingest-tests` as commit-scoped historical ingestion rather than the default workspace-validation path.
- **Late local dependency resolution now survives sync order and resolves cross-file imports**: DevQL current-state writes no longer rely on prepare-time visibility to resolve repo-local dependency edges. An authoritative post-sync reconciliation pass now resolves unresolved local edges against fully materialized current-state artefacts, refreshes canonical local targets when the target path changes or disappears, and keeps historical edge building aligned through the same shared resolver. This now resolves cross-file local calls, references, relationships, and imports across Rust, TS/JS, Python, Java, and C#, including Ruff-style Rust `super::super::...` paths and Rust grouped `use ...::{...}` imports. Go same-package refs continue to resolve, while Go imported package-path edges remain intentionally unresolved until module-path facts from `go.mod` are available to the late resolver.

## [0.0.14] - 2026-04-12

### Added

- **Shared DevQL task orchestration for sync and ingest**: replaced the sync-only daemon queue with a generic persisted DevQL task subsystem that manages both sync and ingest work. The daemon now stores typed task records and repo control state, migrates legacy sync queue documents into the new task state on recovery, supports repo-scoped pause/resume plus queued-task cancellation, and schedules independent per-repo `sync` and `ingest` lanes so one sync and one ingest can run concurrently for the same repository. Ingest correctness remains owned by `commit_ingest_ledger` and branch watermarks rather than orchestration state.
- **Repo-bound daemon config binding for safe capture**: added a local-only `[daemon].config_path` binding in `.bitloops.local.toml`, wired `bitloops init` to store the running daemon's canonicalised config path as the repo binding, and added the `x-bitloops-daemon-binding` request header so repo-scoped daemon requests carry a stable binding identifier derived from that config path.
- **Daemon-owned embeddings bootstrap task and gate state**: added a first-class `EmbeddingsBootstrap` DevQL task with persisted task records, progress phases, queue visibility, CLI rendering, and a daemon-owned `embeddings_bootstrap_state` runtime document keyed by daemon config path. Managed runtime bootstrap now reports explicit phases for config preparation, release resolution, runtime download/extraction, runtime command rewrite, and profile warm-up, while enrichment status now exposes whether embedding work is blocked on bootstrap readiness and which bootstrap task currently owns the gate.
- **Host-managed repo workplane and semantic-clones mailboxes**: added a repo-scoped workplane in the runtime repo with declared `cursor` and `job` mailboxes, row-based persistence, host-owned enqueue/claim/retry semantics, and capability-scoped mailbox registration through the host gateway. `semantic_clones` is the first migrated pack and now declares `semantic_clones.inbound.current_state`, `semantic_clones.summary_refresh`, `semantic_clones.embedding.code`, `semantic_clones.embedding.summary`, and `semantic_clones.clone_rebuild`, replacing the old split between the current-state scheduler and the daemon-wide enrichment queue document.
- **Deferred semantic-clones mailbox intent and backlog compaction**: added repo-scoped mailbox intent persistence for the semantic-clones deferred pipeline so `bitloops init --install-default-daemon` can opt a repository into durable summary, embedding, and clone-rebuild mailboxes before providers are ready. Pending artefact mailboxes now compact into repo-backfill jobs instead of being dropped when blocked backlogs become large or old, while completed and failed mailbox history is retained with bounded pruning.
- **Shared hot local embeddings IPC sessions**: local `bitloops_embeddings_ipc` sessions are now shared process-wide per runtime/model configuration instead of being recreated per embedding job, with idle eviction after 60 seconds so the managed embeddings runtime stays warm while work is flowing without remaining resident indefinitely.

### Changed

- **DevQL GraphQL and CLI surfaces are now task-first**: removed the sync-only GraphQL task fields and the direct/local ingest execution path, and switched the public interface to generic tasks. GraphQL now exposes `enqueueTask`, `task`, `tasks`, `taskQueue`, `pauseTaskQueue`, `resumeTaskQueue`, `cancelTask`, and `taskProgress`, while the CLI now uses `bitloops devql tasks enqueue|watch|status|list|pause|resume|cancel`. `bitloops init`, `bitloops status`, `bitloops daemon status`, and existing sync producers now route through the shared task coordinator, and the checked-in SDL snapshots were refreshed for the new schema.
- **Repo-scoped daemon-backed writes now resolve config strictly from the repo binding**: interaction spool opening, repo runtime-store access, repo-scoped DevQL GraphQL/SDL/WebSocket requests, and other daemon-backed write paths now use `BITLOOPS_DAEMON_CONFIG_PATH_OVERRIDE` first and the repo's local daemon binding second. They no longer fall back to the nearest `config.toml` or the default global daemon config for capture, queue, spool, or turn-data writes.
- **Hook and listener-triggered DevQL work now spools into the repo runtime store before the daemon queue**: `post-commit`, `post-merge`, `pre-push`, post-checkout branch seeding, and watcher-triggered sync capture no longer depend on an active daemon at hook time. They now persist durable producer jobs in the repo runtime database, and the daemon task worker recovers interrupted producer rows on startup, drains them ahead of the shared DevQL task queue, and replays the work into the normal sync/ingest pipeline or the extracted hook-side execution helpers once the daemon is available again.
- **Embeddings bootstrap and semantic-clones enrichment are now asynchronous and mailbox-backed**: `bitloops init --install-default-daemon` now enqueues embeddings bootstrap before any init-triggered sync, prints the bootstrap task id/phase, and continues into sync and ingest without waiting for the runtime download or profile warm-up. `bitloops enable --install-embeddings`, `bitloops embeddings install`, and `bitloops embeddings pull` now enqueue and watch the same daemon-owned bootstrap task. Sync and ingest now fan semantic-clones work into repo-scoped workplane mailboxes instead of the old daemon enrichment queue, embedding and clone-rebuild mailboxes stay pending behind the bootstrap gate until the runtime is ready, and workplane-driven embedding jobs now stay truly incremental instead of escalating into whole-repo refresh work.
- **Provider readiness no longer controls whether semantic-clones work is produced**: once a repository has explicitly opted into the deferred semantic-clones pipeline, sync and ingest now enqueue summary, code-embedding, summary-embedding, and clone-rebuild mailbox work regardless of whether providers are configured, downloaded, or warm. Provider and bootstrap readiness now only block claim-time execution, so work is preserved durably and can drain later when providers become available.
- **Local embeddings bootstrap now stages config changes until the runtime is actually usable**: daemon-managed local embeddings setup now writes the live `local_code` bindings only after the managed runtime download/install has completed and the profile warm-up has succeeded. Bootstrap warm-up uses its own larger first-start timeout budget, while steady-state mailbox execution continues to use the normal runtime request timeout.
- **Semantic-clones code and summary embedding channels are now separated**: code embeddings no longer include summary text in their input and can proceed independently of summary refresh, while summary embeddings remain their own mailbox-backed representation derived from the authoritative summary text.
- **`bitloops init --install-default-daemon` onboarding now shows concurrent setup progress**: the initial sync/ingest confirmations now appear before the setup-complete handoff, which now includes Bitloops branding and the dashboard URL. During onboarding, `init` now renders two live progress lanes: the top lane tracks sync and then ingest, while the bottom lane tracks embeddings bootstrap and then the embedding backlog queue once the runtime is ready. The renderer now also shows a short checklist of current branch analysis, git-history analysis, and local embeddings work, with human-readable units for paths, commits, artefacts, and managed-runtime download size.

### Fixed

- **Cross-project daemon/config misrouting**: Bitloops now rejects repo-scoped daemon requests whose repo binding does not match the running daemon's config path, with a clear `bitloops init` rebind error. When repo binding information is missing or invalid, hook capture now warns and skips instead of guessing another config and risking writes into the wrong project's stores.
- **Capability-event current-state consumer scheduling and queue recovery**: successful empty sync generations now still schedule current-state consumers, no-op completions preserve the existing cursor instead of regressing it, retryable task requeues clear stale `started_at` markers, queue snapshots derive their latest action/timestamp from persisted run state, removed artefacts are stored with an explicit placeholder name, and the sync orchestrator/runtime schema wiring no longer carries dead state or needless per-open formatting overhead.
- **QAT task helpers and bundled runner reliability**: the QAT support helpers now use the current `bitloops devql tasks ...` command surface and persisted task queue schema, and the checked-in `cargo qat` alias now runs the bundled ignored acceptance journey through `cargo-nextest --no-capture` so long daemon-backed runs stream progress reliably.
- **Local embeddings bootstrap and runtime recovery edge cases**: standalone embeddings IPC readiness is now bounded by a real per-request timeout path that kills and recreates wedged child processes instead of letting the session remain poisoned. Bootstrap warm-up no longer fails after 60 seconds just because first-start model load is slower than the steady-state startup timeout, and the daemon now avoids publishing local embeddings config before the managed runtime is actually installed and ready.
