# The Evidence-Driven Development Manifesto

## Closing the Loop Left Open by Spec-Driven Development

Spec-Driven Development made the right first move.

It put intent before implementation. It recognised that software should not begin with code, but with a clear account of what the system is meant to do, how it should behave, what constraints it must respect, and what outcomes it should produce.

That was necessary.

It was not sufficient.

A specification describes intended reality. It does not create reality. It does not prove that the implementation satisfies the intent. It does not know when code has drifted, when tests have gone stale, when architecture has decayed, when production behaviour contradicts the design, or when an AI agent is reasoning from an outdated assumption.

Spec-Driven Development opens the loop.

Evidence-Driven Development closes it.

Spec-Driven Development solved the blank-page problem. It gave developers and agents a structured starting point. It reduced ambiguity before code was written. It helped teams move from vague prompts, tickets, and conversations to explicit direction.

But software does not fail only because we lacked a specification.

Software fails because the relationship between specification and reality decays.

The spec says one thing. The code does another. The tests prove a narrower claim than the team believes. The documentation describes an older system. The architecture has drifted. The production system behaves differently under real conditions. The agent retrieves a stale artefact and treats it as truth.

The existence of a specification is not evidence that the system satisfies it.

A spec is a source of intent, not a source of truth.

The system is the truth. Evidence is how we know what the truth is.

Evidence-Driven Development is the discipline of maintaining justified confidence in software systems. Specifications define what should be true. Implementation attempts to make it true. Evidence establishes whether we are justified in believing it is true. Change invalidates confidence. Re-evaluation restores it.

Development is not complete when code has been generated.

It is not complete when the implementation appears to match the spec.

It is not even complete when tests pass.

Development is complete when justified confidence has been restored.

We believe that specifications are necessary, but insufficient.

We believe that a specification is a claim about desired reality, not proof of actual reality.

We believe that implementation is not fulfilment.

We believe that tests are evidence, but evidence is more than tests.

We believe that every meaningful claim about a system should be supported by fresh, traceable evidence.

We believe that evidence must have provenance: where it came from, when it was produced, what it supports, and under what assumptions it remains valid.

We believe that evidence expires.

We believe that every change is a possible invalidation event.

We believe that stale evidence is not reassurance. It is risk disguised as confidence.

We believe that contradictions should be surfaced, not hidden behind green builds, plausible explanations, or authoritative documents.

We believe that confidence must be earned, not assumed.

We believe that AI agents need more than context. They need claims, artefacts, provenance, dependencies, contradictions, freshness, invalidation rules, and confidence.

We believe that the future of software engineering depends not merely on generating code, but on maintaining truth.

Evidence-Driven Development does not reject specifications. It makes specifications accountable.

It does not reject tests. It gives tests a clearer role: tests are evidence linked to claims, system states, and change.

It does not replace human judgement. It makes judgement explicit, inspectable, and open to revision.

It does not treat confidence as a feeling. It treats confidence as a maintained property of the development process.

Spec-Driven Development asks: what should we build?

Evidence-Driven Development asks: what are we justified in believing about what we have built?

Spec-Driven Development gave us a better way to start.

Evidence-Driven Development gives us a better way to know.

Spec-Driven Development solved the blank-page problem.

Evidence-Driven Development solves the truth problem.
